**เวอร์ชันโปรแกรม:** 1.0.0  
**เวอร์ชันโครงสร้างคอนฟิก (configVersion):** 1.0

## 1. ภาพรวมโปรแกรม (Overview)
**DataBridge** เป็นโปรแกรมประเภท Command-Line Interface (CLI) ที่รันผ่านหน้าต่างคำสั่ง (Terminal) ออกแบบมาสำหรับการย้ายหรือซิงโครไนซ์ข้อมูลจากฐานข้อมูลต้นทาง (Source) ไปยังตารางฐานข้อมูลปลายทาง (Target) 

โปรแกรมถูกคอมไพล์สำเร็จรูปให้อยู่ในรูปของไฟล์ทำงานเดี่ยว (**Single Executable Binary**) ทำให้ทีมผู้ใช้งานสามารถนำไปรันบนเครื่องเซิร์ฟเวอร์ใด ๆ ได้ทันทีโดยไม่จำเป็นต้องติดตั้ง Node.js, Bun หรือไลบรารีอื่นเพิ่มเติม

### ฐานข้อมูลที่รองรับ (ข้ามระบบได้อิสระ):
* **MySQL / MariaDB**
* **Microsoft SQL Server (MSSQL)**
* **PostgreSQL**

## 2. โครงสร้างไฟล์ในชุดแพ็คเกจส่งมอบ (Deliverables)
ชุดโปรแกรมที่พร้อมใช้งานสำหรับทีมงานจะอยู่ในรูปแบบโฟลเดอร์แจกจ่าย ดังนี้:
```
DataBridge-v1.0.0/
├── DataBridge.exe   <- ไฟล์โปรแกรมสำหรับรันบน Windows (หรือชื่อ DataBridge สำหรับ Linux)
├── .env             <- ไฟล์ตั้งค่าระบบและการเชื่อมต่อเครือข่ายฐานข้อมูล
├── config.json      <- ไฟล์กำหนดเงื่อนไขตารางคิวรี และเงื่อนไขการย้ายข้อมูลย่อย
└── README.md        <- คู่มือการใช้งานฉบับนี้ (ภาษาไทย)
```

*หมายเหตุ: โฟลเดอร์ `logs/` จะถูกสร้างขึ้นโดยอัตโนมัติเมื่อโปรแกรมรันครั้งแรก*

## 3. การตั้งค่าตัวแปรระบบ (.env)
ไฟล์ `.env` ใช้เก็บพารามิเตอร์การเชื่อมต่อเครือข่าย ความปลอดภัย และพฤติกรรมในระดับปฏิบัติการ:

| ชื่อตัวแปร | ค่าที่รองรับ | คำอธิบาย |
|---|---|---|
| `SOURCE_DB_TYPE` | `mysql`, `mssql`, `postgresql` | ประเภทของฐานข้อมูลต้นทาง |
| `SOURCE_DB_HOST` | ไอพี หรือ โฮสต์เนม | ที่อยู่เซิร์ฟเวอร์ฐานข้อมูลต้นทาง |
| `SOURCE_DB_PORT` | ตัวเลขพอร์ต (เช่น 3306, 1433, 5432) | พอร์ตเชื่อมต่อของต้นทาง |
| `SOURCE_DB_USER` | ชื่อผู้ใช้งาน | Username สำหรับต่อเข้าฐานข้อมูลต้นทาง |
| `SOURCE_DB_PASSWORD` | รหัสผ่านผู้ใช้ | Password สำหรับต่อเข้าฐานข้อมูลต้นทาง |
| `SOURCE_DB_CONNECTION_LIMIT` | ตัวเลขจำนวน (เริ่มต้น: 5) | จำนวนช่องทางเชื่อมต่อสูงสุดของต้นทาง (Connection Pool Limit) |
| `TARGET_DB_TYPE` / `HOST` / `PORT` / `USER` / `PASSWORD` | เหมือนด้านบน | ข้อมูลการเชื่อมต่อฝั่งปลายทาง (Target) |
| `TARGET_DB_CONNECTION_LIMIT` | ตัวเลขจำนวน (เริ่มต้น: 5) | จำนวนช่องทางเชื่อมต่อสูงสุดของปลายทาง (Connection Pool Limit) |
| `WRITE_MODE` | `insertOnly`, `deleteThenInsert`, `upsert`, `truncateThenInsert` | โหมดการเขียนข้อมูลปลายทาง (ดูรายละเอียดบทที่ 5) |
| `BATCH_SIZE` | ตัวเลขจำนวนแถว (เริ่มต้น: 500) | จำนวนข้อมูลที่จะสตรีมและบันทึกต่อรอบ |
| `MAX_RETRIES` | ตัวเลขจำนวนรอบ (เริ่มต้น: 5) | จำนวนรอบที่จะลองทำซ้ำเมื่อเกิดการเชื่อมต่อขาดหาย |
| `RETRY_BACKOFF` | `exponential`, `fixed` | กลยุทธ์หน่วงเวลารอรอบทำซ้ำ (ทวีคูณ หรือ คงที่) |
| `BATCH_CONCURRENCY_LIMIT`| ตัวเลขจำนวน Batch (เริ่มต้น: 4) | ขีดจำกัดในการเขียนข้อมูลขึ้นฐานข้อมูลขนานพร้อมกัน |
| `PRECHECK_ENABLED` | `true`, `false` | เปิด/ปิดระบบตรวจสอบก่อนรันจริง |

> [!WARNING]
> **คำเตือนด้านความปลอดภัย (Credential Security):**
> รหัสผ่านในการต่อฐานข้อมูลใน `.env` ถูกเก็บไว้ในรูปแบบข้อความดิบ (Plain-text) โดยเจตนาเพื่อความสะดวกในการจัดการ โปรแกรมนี้ห้ามนำไปรันบนเครือข่ายสาธารณะภายนอก และ **ห้ามทำการ Commit ไฟล์ `.env` ขึ้นสู่ Git สาธารณะเด็ดขาด**

## 4. โครงสร้างกำหนดงานย้ายข้อมูล (config.json)
ไฟล์ `config.json` จะระบุเนื้อหางานย้ายข้อมูล โดย 1 ไฟล์ตั้งค่าจะรองรับการย้ายงาน "1 คู่ตาราง (1 Table Pair)" เท่านั้น:

### รูปแบบโครงสร้างคอนฟิก (Version 1.0)
```json
{
  "configVersion": "1.0",
  "description": "คำอธิบายงานย้ายข้อมูลโดยสังเขป",
  "source": {
    "database": "ชื่อฐานข้อมูลต้นทาง",
    "table": "ชื่อตารางต้นทาง",
    "query": "SELECT * FROM ตาราง WHERE store_id = :storeId AND update_date = :updateDate",
    "params": [
      { "name": "storeId", "type": "string", "value": "S001" },
      { "name": "updateDate", "type": "date", "value": "2026-07-17" }
    ]
  },
  "target": {
    "database": "ชื่อฐานข้อมูลปลายทาง",
    "table": "ชื่อตารางปลายทาง",
    "deleteCondition": {
      "conditions": [
        { "column": "STORE_ID", "operator": "=", "value": "$ref:storeId" }
      ]
    },
    "upsertKeys": []
  }
}
```

### ฟังก์ชันพิเศษพารามิเตอร์ลิงก์เงื่อนไข (`$ref:`)
* ในส่วนของ `target.deleteCondition` ท่านสามารถพิมพ์ค่า `value` เป็นรูปแบบ `"$ref:ชื่อตัวแปร"` (เช่น `"$ref:storeId"`) เพื่อสั่งให้โปรแกรมดึงค่าตัวแปรจากกล่อง `source.params` ฝั่งซ้ายมาใช้งานลบฝั่งขวาโดยอัตโนมัติ ช่วยลดความเสี่ยงจากการพิมพ์ค่าตัวเลข/ตัวอักษรซ้ำซ้อนแล้วพิมพ์ผิดพลาด

### ตัวดำเนินการเปรียบเทียบ (Operators) ที่รองรับ:
* `=`, `!=`, `<`, `<=`, `>`, `>=`
* `IN` (ตัวอย่างค่า: `["S001", "S002"]`)
* `BETWEEN` (ตัวอย่างค่า: `["2026-07-01", "2026-07-17"]`)
* `LIKE` (ตัวอย่างค่า: `"S0%"`)

## 5. รายละเอียดโหมดการเขียนข้อมูล (Write Modes)
โหมดการเขียนข้อมูลปลายทางกำหนดผ่านค่าตัวแปร `WRITE_MODE` ใน `.env` โดยมีพฤติกรรมและความต้องการในไฟล์คอนฟิกที่ต่างกันดังนี้:

1. **`insertOnly`:**
   * **พฤติกรรม:** นำเข้าข้อมูลต้นทางไปเขียนต่อท้ายที่ปลายทางทันทีโดยไม่มีการยุ่งเกี่ยวกับข้อมูลเก่า
   * **ข้อกำหนด:** ไฟล์ `config.json` ไม่จำเป็นต้องระบุเงื่อนไขการลบหรือคีย์หลักใด ๆ

2. **`deleteThenInsert`:**
   * **พฤติกรรม:** ทำการลบข้อมูลเดิมที่ตารางปลายทางตามเงื่อนไขที่กำหนดไว้ใน `deleteCondition` ก่อน จากนั้นจึงนำเข้าข้อมูลใหม่ทั้งหมดที่สตรีมมาเขียน
   * **ข้อกำหนด:** **ต้องระบุ** ค่าเงื่อนไขในกล่อง `target.deleteCondition.conditions` หากไม่มีโปรแกรมจะขัดข้องทันทีก่อนเริ่มงาน

3. **`upsert`:**
   * **พฤติกรรม:** ตรวจเช็คข้อมูลด้วยคีย์หลัก หากมีอยู่แล้วให้ทำการอัปเดตข้อมูลแถวนั้น (Update) หากไม่มีให้ทำการเขียนข้อมูลแถวใหม่เพิ่ม (Insert)
   * **ข้อกำหนด:** **ต้องระบุ** คอลัมน์คีย์หลักที่ระบุความเป็นเอกลักษณ์ในอาเรย์ `target.upsertKeys` (เช่น `["ID"]` หรือ `["STORE_ID", "ITEM_CODE"]`)

4. **`truncateThenInsert`:**
   * **พฤติกรรม:** ล้างข้อมูลเก่าทั้งหมดในตารางปลายทางอย่างราบคาบ ก่อนที่จะนำเข้าข้อมูลใหม่ทั้งหมด
   * **ข้อกำหนดทางเทคนิค:** เพื่อความปลอดภัยในการกู้ข้อมูลเมื่อโปรแกรมแครช โปรแกรมจะใช้คำสั่งลบแบบ `DELETE FROM` (แทนคำสั่ง Truncate ที่คอมมิตอัตโนมัติ) เพื่อให้สามารถทำการ Rollback กู้ตารางปลายทางกลับมาเป็นปกติได้หากการซิงค์ล้มเหลวระหว่างทาง

> [!IMPORTANT]
> **ความปลอดภัยระดับธุรกรรม (Transaction Lock):**
> ทุกโหมดการเขียน (ยกเว้น insertOnly) จะทำงานภายใต้ธุรกรรมเดียวกัน (Single Database Transaction) ครอบคลุมทั้งขั้นตอนล้างและเขียน หากการเขียนแถวข้อมูลล้มเหลวแม้แต่ Batch เดียว ข้อมูลปลายทางจะถูก Rollback ย้อนคืนเสมือนไม่มีอะไรเกิดขึ้น แต่พึงระวังว่าการทำงานกับข้อมูลหลายร้อยล้านแถวอาจใช้เวลานานและมีการล็อคตารางปลายทางชั่วคราว

## 6. วิธีการใช้งานโปรแกรม (Usage Guide)

### 6.1 ตรวจสอบเวอร์ชันโปรแกรม
ใช้คำสั่งแสดงรุ่นของตัวซอฟต์แวร์ไบนารี:
```powershell
# บน Windows
DataBridge.exe --version

# บน Linux
./DataBridge --version
```

### 6.2 สั่งรันกระบวนการย้ายข้อมูล (Migration Run)
ระบุพาธไฟล์งาน `.json` ผ่านทางตัวเลือกคอนฟิก:
```powershell
# บน Windows
DataBridge.exe --config=./config.json

# บน Linux
./DataBridge -c ./config.json
```

## 7. ระบบรายงานบันทึกประวัติ (Logging)
ทุกขั้นตอนการทำงาน ความขัดข้อง และความคืบหน้าจะถูกรายงานออกมาผ่าน 2 รูปแบบพร้อมกัน:
1. **Console:** แสดงผลลัพธ์แยกสีตามระดับเหตุการณ์บนหน้าจอคำสั่ง
2. **Daily rotated files:** บันทึกประวัติลงเป็นไฟล์ `.log` แยกรายวันในรูปแบบ JSON เพื่อให้อ่านเข้าใจง่ายและเก็บประวัติถาวร (เช่น `logs/2026-07-17.log`)

หากโปรแกรมเกิดขัดข้อง ท่านสามารถเข้าไปยังโฟลเดอร์ `logs` เพื่อเปิดไฟล์เหตุการณ์วันดังกล่าวเพื่อตรวจสอบสาเหตุอย่างละเอียดได้ทันที
